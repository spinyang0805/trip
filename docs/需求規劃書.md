## 2. 新需求規格

### 需求 1:從 Google 地圖點擊加入行程

**現況**:只能在左側清單打字新增,靠 Geocoder 找座標。

**目標**:地圖上直接點 POI(店家、景點圖標)→ 彈出資訊卡 → 一鍵加入指定天數。

**規格**
- 地圖 `click` 事件帶 `placeId` 時(點到 POI),開啟自製資訊卡(禁用預設 InfoWindow)
- 資訊卡顯示:名稱、評分、地址、照片縮圖(Places Details API:`name, rating, formatted_address, photos, geometry, place_id`)
- 資訊卡底部:「加入 Day N」下拉 + 確認鈕 → 寫入該天 spots,自動帶入名稱與座標,`placeId` 一併儲存
- 點擊空白處(無 placeId)→ 反查 Geocoder,提供「以此座標新增自訂景點」
- 已在行程內的 POI,資訊卡顯示「已在 Day N」標記

**技術備註**:Places Details 走 JS API(`google.maps.places.Place`),沿用現有 API key,需在 Cloud Console 確認 Places API (New) 已啟用。

---

### 需求 2:自動行車路徑

**現況**:已有——每天景點間以 DirectionsService 畫路線、分色、可開 Google Maps 導航。

**規格(強化)**
- 保留現有行為,補上:
  - 每段路線顯示**行車時間與距離**(DirectionsService 回應已含,目前丟棄未用)
  - 天卡片頂部顯示該日總車程時間
  - 路線計算失敗的路段(如跨海無路)以虛線直線 fallback(現有)+ toast 提示
- 順序變更(需求 4)後路線**自動重算**

**技術備註**:Google 已標示 DirectionsService 為 legacy(2026/02),短期可續用;規劃書先記錄,`Routes API` 遷移列入後續(§6)。

---

### 需求 3:每日照片上傳(兩類)

**目標**:每天行程可掛照片,分兩個獨立區塊:

| 類型 | 用途 | 範例 |
|---|---|---|
| 規劃照片 `planning` | 行前參考 | 店家菜單、想看的風景、攻略截圖 |
| 遊記照片 `memory` | 出遊實拍 | 當天拍的照片 |

**規格**
- 每個天卡片內兩個分頁籤:「📋 規劃」「📸 回憶」
- 上傳:選檔或手機直接拍照(`<input type="file" accept="image/*" capture>`)
- 縮圖牆顯示,點擊放大(lightbox),可刪除(上傳者或管理員)
- 每張照片記錄:上傳者、時間、可選說明文字
- client 端壓縮(canvas 縮到長邊 ≤1600px、JPEG 品質 0.8)再上傳,控制流量與容量

**儲存方案決策(需拍板)**

| 方案 | 費用 | 隱私 | 備註 |
|---|---|---|---|
| A. Firebase Storage(建議) | 需升級 Blaze(綁信用卡),有免費額度(5GB 儲存),家庭用量幾乎 $0 | ✅ Storage Rules 可做成員制存取 | 與現有 Firebase 整合最順,**私人文件(需求 7)只有此方案能做存取控制** |
| B. Cloudinary 免費方案 | $0(25 credits/月) | ⚠️ unsigned upload,URL 公開(難猜但無權限控制) | 純前端可用,適合照片、不適合私人文件 |
| C. Base64 存 RTDB | $0 | ✅ | RTDB 免費上限 1GB,幾十張照片就滿,**不可行** |

> **建議:方案 A**。Blaze 有免費額度,設每月預算警示($1)防呆;需求 7 的私人文件無論如何都需要它。若不願綁卡,退而求其次照片用 B、需求 7 改存外部連結(Google Drive 連結貼上)。

**資料模型**
```
photos/{projectId}/{dayId}/{photoId}: {
  kind: "planning" | "memory",
  url, storagePath, caption, uploadedBy, uploadedAt
}
```
檔案路徑:`projects/{projectId}/days/{dayId}/{photoId}.jpg`

### 需求 4:景點卡片排序(手機友善)

**現況**:桌面 HTML5 drag & drop,手機無法用。

**規格**
- 每張景點卡片右側加「▲ / ▼」按鈕:上移一位 / 下移一位
- 移動後**序號圓圈立即重編**(1, 2, 3…),路線與費用小計自動重算
- 保留桌面拖曳;手機以按鈕為主(touch 拖曳排序列入後續強化)
- 跨天移動:卡片選單提供「移至 Day N」

**技術備註**:現有 spots 以物件 key 排序,需加明確 `order` 欄位(整數),`getSpotsArray` 改按 `order` 排;既有資料遷移:首次讀取無 `order` 時按現行順序補號寫回。

---

### 需求 5:天數展開 / 收合

**目標**:天數多時畫面過長、手機字太小。

**規格**
- 天卡片標題列(Day N + 日期 + 當日費用)常駐,點擊即展開/收合景點清單
- 收合狀態顯示摘要:景點數、總費用、起訖景點名
- 預設:**全部收合,僅展開第一天**(或今天,若在旅程期間)
- 收合狀態存 `localStorage`(個人偏好,不進 DB)
- 「全部展開 / 全部收合」切換鈕
- 同步調整手機版字級與間距(標題列加大、觸控目標 ≥44px)

---

### 需求 6:景點詳情頁 = 共用開源景點庫

**目標**:景點不再只是行程裡的一行字。每個景點可進入「詳情編輯」,內容成為**跨專案共用的景點文章**,像開源 wiki:任何專案可搜尋、引用、共同編輯。

**規格**

**a. 共用景點庫(新節點 `pois/`)**
```
pois/{poiId}: {
  name, nameAliases: [],           // 搜尋用
  placeId,                          // Google place_id(可選,去重用)
  lat, lng, address,
  category,                         // 沿用 12 種類型
  article,                          // 文章內容(Markdown 純文字)
  photos: { {photoId}: {url, caption, uploadedBy} },   // 公用照片
  links: [],                        // 官網、tabelog 等
  createdBy, updatedBy, updatedAt,
  usedByCount                       // 被引用次數
}
```

**b. 行程景點與 POI 的關係**
- `spots/{spotId}` 加 `poiId` 欄位(可為空 = 純自訂景點)
- 景點卡片點「📖 詳情」→ 開啟詳情面板(modal 或獨立頁 `poi.html?id=`)
- 新增景點時:先搜尋 `pois/`(名稱 + placeId),命中 → 引用並帶入座標照片;未命中 → 建立新 POI

**c. 詳情頁內容區塊**
1. 基本資訊(名稱、分類、地址、地圖小窗)
2. 文章(所有 approved 使用者可編輯,存最後編輯者;簡易版本記錄:保留最近 10 版於 `poisHistory/`)
3. 公用照片牆(菜單、環境照——與需求 3 的「規劃照片」可一鍵複製過來)
4. **私人文件區(見需求 7,僅本專案成員可見)**
5. 引用資訊:「被 N 個專案使用」

**d. 搜尋**
- 詳情頁與新增景點框提供搜尋:RTDB 全量拉 `pois` 的 `name/nameAliases/category` 建前端索引(家庭規模數百筆內可行),關鍵字比對 + 分類篩選

**e. 權限**
- 讀:所有 approved 使用者
- 寫(文章/照片):所有 approved 使用者(開源共編精神)
- 刪 POI:僅管理員

---

### 需求 7:景點私人文件區(專案私有)

**目標**:住宿訂單、機票 PDF、預約確認信等,掛在景點或專案下,**只有該專案成員可見**,絕不進共用景點庫。

**規格**
- 掛載位置兩種:
  - 景點詳情頁「🔒 私人文件」區塊(訂房單掛飯店景點下)
  - 專案層「文件夾」頁籤(機票等不屬於單一景點的檔案)
- 支援格式:PDF、圖片、常見文件(≤10MB/檔)
- 顯示:檔名、上傳者、時間、下載/預覽(PDF 用瀏覽器內建 viewer)
- 刪除:上傳者或管理員

**資料模型**
```
docs/{projectId}/{docId}: {
  spotPoiId,            // 可為空 = 專案層文件
  fileName, url, storagePath, size, mime,
  uploadedBy, uploadedAt
}
```
檔案路徑:`projects/{projectId}/docs/{docId}_{fileName}`

**權限(關鍵)**
- RTDB `docs/{pid}`:僅該專案 owner/成員/管理員可讀寫(同 `trips` 規則)
- Firebase Storage Rules:`projects/{pid}/**` 比照——**必須用方案 A(Firebase Storage)才能實現**;共用 POI 照片放 `pois/**` 路徑,approved 使用者可讀

## 3. 安全規則增補(隨功能一起發布)

```
pois:        讀=approved;寫=approved;刪=admin
poisHistory: 讀=approved;寫=approved(append)
photos/{pid}: 讀寫=該專案成員(同 trips 規則)
docs/{pid}:   讀寫=該專案成員(同 trips 規則)
```
Storage Rules 對應:`pois/**`(approved 可讀寫)、`projects/{pid}/**`(成員限定,經由 RTDB 成員名單驗證需搭配 custom claims 或以 metadata 驗證——實作時採用「上傳走成員檢查後的前端 + 讀取用下載 URL(帶 token)」的簡化模型,細節實作時定)。

## 5. 實作階段規劃

| 階段 | 內容 | 相依 |
|---|---|---|
| **P1 基礎重構** | spots 加 `order` 欄位與遷移、排序按鈕(需求 4)、天數收合(需求 5) | 無外部相依,立即可做 |
| **P2 地圖強化** | POI 點擊加入(需求 1)、路線時間距離(需求 2) | 確認 Places API 啟用 |
| **P3 儲存開通** | Firebase 升級 Blaze + 預算警示、Storage Rules | **需你操作**(綁卡) |
| **P4 照片** | 每日兩類照片(需求 3) | P3 |
| **P5 景點庫** | `pois/` 共用庫、詳情頁、搜尋引用(需求 6) | P1 |
| **P6 私人文件** | 文件上傳與權限(需求 7) | P3、P5 |

> P1、P2、P5 不需要 Blaze,可先行;P3 只是 Console 操作十分鐘。
